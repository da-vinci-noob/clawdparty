## Context

This is **the Rails backend stream** (`docs/PLAN.md §10`). The `freeze-interface-contracts` change has frozen four capabilities (`event-envelope`, `sidecar-protocol`, `http-api-contract`, `contracts-package`); this change is the first **consumer** that ships running behavior. It must stand up enough Rails to satisfy the W1 milestone — *"Rails + sidecar can replay the fixture end-to-end; a watching browser sees it"* — without building the W2/W3 surface (run orchestration, sidecar HTTP client, file/diff APIs, changeset service).

Constraints inherited from `CLAUDE.md` and `docs/PLAN.md`:
- Postgres 18 with **3 logical databases** (primary, queue, cable) for Solid Queue + Solid Cable. Runs in Docker via `bin/start` (from the `dev-docker-compose` change).
- The two load-bearing invariants are enforced **at the database**, not in Ruby: one active run per session (partial unique index) and idempotent ingest (`[ai_run_id, seq]` unique index).
- Follow the PLAN §16 conventions (service POROs, `rescue_from` JSON error rendering, the cable connection auth shape, one minimal factory per model) and **skip** scale machinery a small MVP doesn't need: a command/interactor framework, a row/attribute-level authorization framework (the 4-role `SessionPolicy` PORO is right-sized), modular package boundaries, and the Redis cable adapter.
- The event stream alone must reconstruct the UI: every mutation appends an event in the same transaction.

The most subtle design pressure is **Decision 9** from the contracts' design: the fake-Claude rake calls `Events::Ingest` directly (not over HTTP), so "real ingest" is only real if broadcast lives **inside** `Events::Ingest` and the `/internal/events` controller stays thin. This shapes where the broadcast call goes and how the controller is tested.

## Goals / Non-Goals

**Goals:**
- Rails 8 API-only scaffold with the Postgres-18 / Solid Queue / Solid Cable / RuboCop / RSpec / FactoryBot / annotaterb toolchain and a green `api` CI job.
- The 8-table data model with enums, the `ai_runs` state machine values, the `events.actor` columns, and the **two load-bearing indexes** enforced at the DB.
- Invite-link auth (SHA-256 digests, role-scoped, expiry/revoke) → signed httpOnly `clawd_uid` cookie, used by both REST and `ApplicationCable::Connection`; `SessionPolicy` 4-role enforcement; cable participantship verification.
- The ingest pipeline: `Events::Ingest` (persist/dedupe/ephemeral-skip/**broadcast-in-service**), `Events::Append` (mutation+event same txn), `SessionChannel`, the **thin** `POST /internal/events` controller + its wire-contract request spec.
- The fake-Claude rake task replaying `sample_run.jsonl` via direct in-process `Events::Ingest`, powering seeding and the happy-path system test.

**Non-Goals:**
- Run orchestration (`Runs::Start`/`Finalize`), the sidecar HTTP client, worktree creation, diff/file APIs, the changeset service (approve/reject) — all W2/W3. W1 only ensures the schema + state-machine values can *represent* these later (e.g. the reject-severs-chaining rule needs `superseded`/`rejected` states to exist).
- Building the SPA or cable client (`web` stream); finalizing per-type event payload schemas (still `pending-spike` in the contract — W1 treats `payload` as opaque JSON).
- Sidecar code of any kind. This stream never imports the Agent SDK.

## Decisions

**1. API-only Rails 8 with three explicit databases.** `config/database.yml` declares `primary`, `queue`, and `cable` connections; `ApplicationRecord` uses `primary`, Solid Queue and Solid Cable point at their own. *Why:* `docs/PLAN.md §2` chose Solid Queue + Solid Cable specifically to avoid Redis; the 3-DB split is the standard Rails 8 layout and keeps queue/cable churn off the primary. Alternative (single DB) was rejected — it couples cable/queue write volume to app data and diverges from the team's `core` muscle memory.

**2. The two invariants live in the schema, asserted by spec.** Migration adds `add_index :ai_runs, :session_id, unique: true, where: "status IN ('queued','running','awaiting_review')"` and `add_index :events, [:ai_run_id, :seq], unique: true`. Models add matching validations only as a courtesy; the DB is the source of truth. *Why:* a Ruby-level check races under concurrent run starts and concurrent ingest batches; only a partial unique index actually prevents two active runs, and only the composite unique index makes ingest idempotent under retries. Specs assert the constraint by attempting the duplicate and expecting `ActiveRecord::RecordNotUnique` (caught and translated, see Decision 6).

**3. `events.actor` is stored as `actor_kind` (enum: claude/user/system) + nullable `actor_participant_id`.** This reconstructs the frozen `event-envelope` actor discriminated union `{ kind: "claude" } | { kind: "user"; id } | { kind: "system" }`: `actor_participant_id` is non-null **iff** `actor_kind == "user"`. The serialized envelope rebuilds the union from these two columns; the participant **id** is stored (names resolved client-side, role never stored on the event). *Why:* matches contract Decision 10 exactly; storing a flat string actor would lose the id and force per-payload stuffing. A DB check constraint (`actor_participant_id IS NOT NULL = (actor_kind = 'user')`) makes the illegal states unrepresentable.

**4. `ai_runs.status` is an enum covering the full frozen state machine** — `queued`, `running`, `awaiting_review`, `approved`, `rejected`, `superseded`, `completed_clean`, `failed`, `interrupted`. W1 only sets a subset in practice (the replay/seed path), but all nine exist so W2 `Runs::Start` and the reject/revise rules can be encoded without a migration. *Why:* the reject-severs-chaining and revise-supersedes rules (`docs/PLAN.md §8`) are correctness rules that need `rejected`/`superseded`/`approved` to exist; adding them now is free and avoids a schema change mid-W2. The state-machine *transitions* are deferred to W2; W1 ships only the value set + the active-run partial index that gates them.

**5. Broadcast lives inside `Events::Ingest`, and `/internal/events` is a thin controller.** `Events::Ingest.call(event_attrs)` does: classify ephemeral-vs-durable → for durable, insert with dedupe-on-`(ai_run_id, seq)` (skip silently on conflict) → **then broadcast the envelope via `SessionChannel.broadcast_to`** (ephemeral types skip persistence but still broadcast). The controller does only `auth → parse batch → Events::Ingest.call(each) → render`. *Why:* this is contracts Decision 9 made concrete — the fake-Claude rake calls `Events::Ingest` directly, so if broadcast were in the controller a direct replay would persist but never broadcast and a watching browser would see nothing, failing the "watchable" milestone. Putting broadcast in the service makes "direct == real" hold. Alternative (broadcast in controller, or a separate `Events::Broadcast` step the rake must also call) was rejected: it duplicates the broadcast call site and is the exact bug Decision 9 warns against.

**6. `Events::Ingest` swallows the dedupe conflict; the wire contract is tested separately.** Ingest rescues `ActiveRecord::RecordNotUnique` on `(ai_run_id, seq)` and returns a "skipped" result rather than raising — so a re-POSTed batch is idempotent. The HTTP wire contract (bearer `SIDECAR_SHARED_SECRET` auth, batch-envelope parsing, rejection of bad/missing auth) is verified by a **focused `/internal/events` request spec**, not by the rake. *Why:* splits "ingestion core" (exercised identically by direct call and HTTP) from "wire contract" (HTTP-only), per contracts Decision 9. `docs/PLAN.md §13` already lists "ingest secret" as a must-test. The controller returns `200` with `{ accepted, skipped }` for a re-POSTed batch; `409` is reserved for run-start conflicts, not this batch endpoint (pinned in the `/internal/events` request spec and the frozen `sidecar-protocol` capability). The model-level idempotency is the load-bearing part.

**7. Invite tokens are stored as SHA-256 digests; the raw token is shown once.** `Invite` stores `token_digest` (SHA-256 of a random token), `role`, optional `expires_at`, and a revoke flag/timestamp. Join: look up by digest, check not-expired and not-revoked, create/find a `User` by display name, create a `Participant` with the invite's role, then set a signed httpOnly `clawd_uid` cookie (no `Secure` flag). *Why:* `docs/PLAN.md §9` — the trusted LAN is the perimeter, tokens gate everything, no passwords. Digesting means a leaked DB row can't be replayed as a token. No `Secure` flag because the LAN is plain HTTP (documented accepted risk).

**8. One cookie authenticates REST and cable; `SessionPolicy` gates REST, the channel re-checks participantship.** `ApplicationCable::Connection` uses the standard ActionCable idiom — `identified_by :current_user` + `find_verified_user` + `reject_unauthorized_connection` — where `find_verified_user` resolves the signed `clawd_uid` cookie. `SessionPolicy` (PORO, 4 roles) is called in every controller action; `SessionChannel#subscribed` independently verifies the connected user is a participant of the session before `stream_for`. *Why:* `docs/PLAN.md §9` + §16 — server enforces, client only hides buttons; cable auth is independent of REST policy so a subscribe can't be smuggled past role checks. The 4-role action matrix is taken verbatim from the frozen `http-api-contract` capability (owner > editor > reviewer > viewer).

**9. The fake-Claude rake reads `packages/contracts/fixtures/sample_run.jsonl` and calls `Events::Ingest` per line, in-process.** It accepts/creates a target session and run, then ingests each fixture event (assigning/honoring `(ai_run_id, seq)`), so it runs with no Puma and no `SIDECAR_SHARED_SECRET`. The same path seeds data and backs the one happy-path system test. *Why:* contracts Decision 9 — server-independent, deterministic, and genuinely real ingest (so it both seeds *and* broadcasts, watchable live). If the fixture is still envelope-only placeholder (spike slip), the rake still works because W1 treats `payload` as opaque.

**10. `messages` and `events` are append-only; every mutation appends an event in the same transaction via `Events::Append`.** `Events::Append` wraps a mutation + the corresponding event insert in one DB transaction (e.g. creating a `Message` also appends a `chat_message` event; a `Participant` join appends `participant_joined`). *Why:* `docs/PLAN.md §5/§6` — the event stream alone must reconstruct the UI, so the event must be atomic with the state change. `Events::Append` is the write-side sibling of `Events::Ingest` (which handles the sidecar/replay read-side); both funnel through the same broadcast.

**11. Contract dependency: the W2 run-start request carries `requested_by` (recorded here so it is not lost).** Run orchestration (`Runs::Start`, the sidecar HTTP client, `POST /runs`) is a W2 non-goal, so there is no run-start spec in this change. But the frozen `sidecar-protocol` `POST /runs` payload now includes `requested_by` — the originating participant id — which the sidecar stamps as `run_started.actor.id`. When W2 builds `Runs::Start`, the request it sends to the sidecar's `POST /runs` MUST include `requested_by` sourced from `ai_runs.requested_by` (the participant who started the run). W1 already provisions the data side: `ai_runs.requested_by` exists in the schema (`rails-data-model`, task 2.2), so W2 only has to forward it. Nothing in this change contradicts `requested_by` flowing to the sidecar.

**12. The `tasks` table is built in W1 even though the task-board UI is cut from the MVP.** The task board is out of MVP scope (`docs/PLAN.md §12`), but `rails-data-model` still builds the `tasks` table (and `tasks.status` enum) now. *This is deliberate over-provisioning* (decided by the user): standing up the table in W1 means restoring the board later needs no migration, and the frozen `task_created`/`task_updated` event names are additive-safe regardless of whether the board ships.

**13. Deferred W2 invariant (recorded so it is not lost): reject severs `claude_session_id` chaining.** After a reject (`git reset --hard HEAD && git clean -fd` in the worktree), the next run MUST NOT resume the old Claude session — its context believes the reverted edits still exist. W1 provisions the data for this (`ai_runs.claude_session_id` column, and the `rejected`/`superseded` states exist in the status enum), but the rule itself is encoded by the future W2 `Runs::Start`/changeset change, NOT here. Only **revise** resumes the session. Recorded here so the W2 proposal must address it.

## Risks / Trade-offs

- **Direct-replay broadcasts nothing (the Decision-9 trap).** If a future refactor moves broadcast into the controller → a watching browser sees nothing during replay. **Mitigation:** broadcast is an explicit requirement of the `event-ingest-pipeline` spec (broadcast-in-service), and the fake-claude-replay spec asserts replay both persists *and* broadcasts; the system test catches a regression.
- **Idempotency relies on the DB constraint, not Ruby.** A `find_or_create` style check would race. **Mitigation:** rely on the composite unique index + rescue `RecordNotUnique`; a request spec re-POSTs a batch and asserts no duplicate row and no error.
- **Cookie without `Secure` on plain-HTTP LAN.** Anyone on the LAN can read the page. **Mitigation:** documented accepted MVP risk (`docs/PLAN.md §9`); every endpoint still requires a valid invite-derived cookie; Tailscale future phase flips this on with no app change.
- **Three-DB Solid setup is fiddly in Docker.** Wrong `database.yml` wiring silently routes cable/queue to the primary. **Mitigation:** explicit per-connection config + a smoke check that Solid Cable/Queue tables live in their own DBs; `bin/start` runs `db:prepare` for all three.
- **State-machine values exist before transitions do.** W1 ships nine `ai_runs` statuses but only the replay subset is set, risking a false sense that orchestration works. **Mitigation:** scope is explicit in the proposal/specs — W1 = schema + active-run index only; transitions are a W2 `Runs::*` change.
- **Enum drift vs the frozen taxonomy.** Event `type` and role names must match the frozen `event-envelope`/`http-api-contract` lists exactly. **Mitigation:** specs reference those capabilities by name rather than re-enumerating; a spec asserts persisted `event_type` values are taxonomy members.

## Migration Plan

This is greenfield — no rollback of existing data. Bring-up order: (1) scaffold Rails + `database.yml` (3 DBs) + Solid Queue/Cable; (2) migrations for the 8 tables + 2 indexes + actor check constraint, `db:prepare`; (3) models + factories + annotaterb; (4) invite-auth + `SessionPolicy` + cable connection; (5) `Events::Append`/`Events::Ingest` + `SessionChannel` + thin `/internal/events` controller; (6) fake-Claude rake + happy-path system test; (7) wire `api` CI job (RuboCop + RSpec). Each step is independently green before the next.

## Resolved (previously open)

- **`/internal/events` status on duplicate batch** → `200` with `{ accepted, skipped }`; `409` is reserved for run-start conflicts (`POST /runs`), not this batch ingest. Pinned in the `/internal/events` request spec and the frozen `sidecar-protocol` capability. Model-level dedupe on `(ai_run_id, seq)` is idempotent regardless of status.
- **`User` identity across re-joins** → join SHALL always create a **distinct `Participant`** for each join, even when the same `User` is found-or-created (e.g. the same display name re-joins). The `User` may be reused, but each human-session-join gets its own `Participant` row, so the participant id — which is what the event-envelope `actor.id` carries — stays unique per join. *Why:* the envelope's `actor` is the participant id, not the user id; reusing a `Participant` across joins would make two distinct join sessions collide on one `actor.id` and muddy presence/last-seen tracking. This is consistent with `invite-auth` (each join creates a `Participant` carrying the invite's role) and `rails-data-model` (the `events` actor stores `actor_participant_id`, never a user id or display name).
