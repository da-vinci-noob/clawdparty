## Context

The `sidecar/` is the one process that wraps `@anthropic-ai/claude-agent-sdk`; nothing else in clawdparty imports the SDK (`CLAUDE.md`, `docs/PLAN.md §7`). This change is the **Week-1 skeleton** of that process — the sidecar stream's "sidecar skeleton: HTTP server, normalizer v1, event POST to Rails" work (`docs/PLAN.md §10`). It exists to satisfy exactly one milestone: **Rails + sidecar can replay the fixture end-to-end** (`docs/PLAN.md §15`). That milestone exercises the *plumbing* (Rails reaches the sidecar; the sidecar normalizes shapes to Contract-1 envelopes and POSTs them idempotently to `/internal/events`), not live Claude execution.

The two seams this change builds against are already **frozen** by `freeze-interface-contracts`: the `event-envelope` capability (envelope fields, the 20 type names + `ai_raw` fallback, `(ai_run_id, seq)` idempotency, ephemeral `ai_text_delta`/`presence_changed`) and the `sidecar-protocol` capability (endpoint signatures in both directions, `SIDECAR_URL`/callback addressing, bearer auth, the worktree/`base_sha` rule). This change **consumes** them — it must not re-derive or contradict them.

Hard timeline constraint from `docs/PLAN.md §10/§14`: the *full* per-type normalizer mapping table can only be written after the **Tuesday SDK spike** (raw SDK shapes are the least-known dependency). So the normalizer ships in two layers: a v1 never-crash skeleton now, the full table later. The route handlers for `POST /runs`/`messages`/`interrupt` are likewise stubs — the runner state machine is Week 2.

## Goals / Non-Goals

**Goals:**
- Stand up the Fastify server (Node 24, TypeScript) on the unpublished `:8787` compose-network address with a working `GET /healthz` and stubs for the three run-control routes.
- Emit the 5s heartbeat to Rails (`POST /internal/sidecar/heartbeat` with `active_run_ids`) per the `sidecar-protocol` contract.
- Ship `normalizer.ts` v1: the only SDK-aware file, producing Contract-1 envelopes, with the never-crash unknown→`ai_raw` rule and raw-SDK fixtures owned here.
- Ship `transport.ts`: batched, idempotent, retry-with-backoff POST to `/internal/events`, ring-buffering while Rails is down, bearer-authed.
- Ship `permissions.ts` as an allow-all `canUseTool` stub (the documented Bash-gating seam).
- Establish the **no-app-owned-credential / no-method-selection** rule for Claude auth, and document the two host caveats.
- Wire the `sidecar` CI job (Biome + `tsc` + Vitest) pinned to Node 24.

**Non-Goals:**
- The `runner.ts` run lifecycle/state machine, the pushable streaming-input iterable, real `query()` execution, and `interrupt()` behavior — Week 2 (route handlers are stubs here).
- Worktree creation — **Rails owns it** (`sidecar-protocol`); the sidecar only receives `cwd`.
- `hooks.ts` (Bash→`terminal_output`, Edit/Write→`file_changed`) and the **full per-type** normalizer mapping table — Week 2 / post-spike.
- The Docker bind-mounts (`~/.claude`, `~/.aws`) and host auth-env passthrough — implemented in `dev-docker-compose`; this change only forbids credential/selection code in the sidecar.
- Crash-recovery semantics on the Rails side (`Sidecar::HealthcheckJob`, boot reconciliation) — a later Week-2 Rails run-orchestration change (NOT `rails-foundation`, whose W1 scope is ingest/model/auth only).

## Decisions

**1. Two-layer normalizer: v1 never-crash skeleton now, full mapping table post-spike.**
v1 implements one hard rule — any SDK message type the normalizer does not recognize becomes an `ai_raw` Contract-1 event, never dropped and never a thrown error. The per-type mapping (deltas→`ai_text_delta`, block-stop→`ai_text`, thinking→`ai_thinking`, tool use→`tool_started`/`tool_finished`/`tool_failed`, result→`run_finished`, etc.) is **pending-spike** and stubbed. *Why:* `docs/PLAN.md §14` makes SDK-shape surprises the top risk and forbids inventing the table before the Tuesday spike ("schemas invented before the spike are fiction"). The never-crash rule is the only normalizer behavior we can correctly commit to before seeing real output, and it is exactly what keeps a future unknown SDK message from taking down a live run. *Alternative rejected:* writing the full table now against guessed shapes — explicitly forbidden by the plan and the contract.

**2. The normalizer is the ONLY file that touches raw SDK message shapes.** Per `CLAUDE.md`/`docs/PLAN.md §7`, `index.ts`/`transport.ts`/`permissions.ts` deal only in Contract-1 envelopes. *Why:* a single SDK-aware file means an SDK version bump or shape surprise is contained to one place; everything downstream is shape-stable. This is an architectural boundary, not a style preference.

**3. Raw-SDK fixtures are owned HERE; `sample_run.jsonl` is owned by the contracts change.** This change checks in *raw* SDK message logs under `sidecar/` as the **input** to normalizer unit tests; the **output** the tests assert is the post-normalization `packages/contracts/fixtures/sample_run.jsonl` owned by `freeze-interface-contracts` (its Decision 6). *Why:* conflating the two fixture sets is the single most likely source of confusion (called out in the contracts design). The normalizer test reads raw-in, asserts contract-envelopes-out, and that asserted output must equal `sample_run.jsonl` — so the two fixture sets are kept in distinct ownership but verified against each other.

**4. Transport: batch + per-event `(ai_run_id, seq)` idempotency + ring-buffer + retry-with-backoff.** `transport.ts` accumulates normalized envelopes, POSTs them in batches to `/internal/events` with the `SIDECAR_SHARED_SECRET` bearer token, and on failure (Rails down / 5xx / network) holds events in an in-memory ring buffer and retries with backoff. *Why:* `docs/PLAN.md §7` crash-recovery requires "Rails restarts → sidecar ring-buffers events + retries with backoff." Safety comes for free from the frozen `event-envelope` idempotency rule — Rails dedupes on `(ai_run_id, seq)`, so re-POSTing a buffered batch can never double-insert. The sidecar assigns the per-run monotonic `seq`; it must NOT carry `seq` across runs (it is scoped to `ai_run_id`). *Trade-off:* an in-memory ring buffer is lost if the sidecar process itself dies — accepted for MVP, because the Claude session JSONL in `~/.claude/projects/` is the durable recovery path (`docs/PLAN.md §7`), not the buffer.

**5. Ephemeral `ai_text_delta` is coalesced (~150ms) and broadcast, never sent through the durable ingest batch.** Per `event-envelope`, `ai_text_delta` and `presence_changed` are ephemeral. The W1 skeleton records this as a constraint on the normalizer/transport boundary (deltas do not get persisted), even though delta *generation* is wired fully in Week 2 with the runner. *Why:* a modest run emits 10–20k deltas; the contract decided ephemerality at the envelope layer and all streams must agree. We bake the classification in now so Week-2 wiring can't accidentally persist deltas.

**6. The three run-control routes are stubs; only `/healthz` is functional in W1.** `POST /runs`, `POST /runs/:id/messages`, `POST /runs/:id/interrupt` validate/parse and return a not-yet-implemented response, wired to the runner in Week 2; `GET /healthz` returns `{ active_run_ids }` (empty in W1, since there is no runner). *Why:* the W1 milestone needs Rails to *reach* the sidecar and the heartbeat/ingest loop to work end-to-end — it does not need live runs. Building the stubs against the frozen `sidecar-protocol` signatures means Week 2 fills bodies without changing the surface. *Alternative rejected:* building the runner now — that is the Week-2 task and the largest source of run-lifecycle bugs (`docs/PLAN.md §14`), deliberately deferred.

**7. `canUseTool` is an allow-all stub.** `permissions.ts` returns allow for every tool. *Why:* MVP approval is changeset-level (review the whole diff), not per-tool (`docs/PLAN.md §1`); per-tool Bash gating is an explicit future phase (`§17`). Keeping the seam as a single named file means later activation is local. The terminal pane is a read-only replay — there is no shell input path anywhere, and this stub does not introduce one.

**8. Claude auth: zero credential code, auth-method-agnostic, SDK auto-detects.** The sidecar contains **no app-owned Anthropic key** and **no code that selects or stores a credential or auth method**. It relies entirely on the host developer's existing login inherited at the process/container level — direct API key, Claude subscription/enterprise OAuth, or Amazon Bedrock all "just work" because the SDK auto-detects in its own precedence order (cloud-provider flag → `ANTHROPIC_AUTH_TOKEN` → `ANTHROPIC_API_KEY` → `CLAUDE_CODE_OAUTH_TOKEN` → `~/.claude` creds). *Why:* `docs/PLAN.md §2` (Claude-auth row) and §9 — a run bills/acts as the host's own login; the app must never own or pick a key, and mounts are read-only so a run can't tamper with host login state. The bind-mount/env *wiring* is `dev-docker-compose`'s job; this change's requirement is purely the absence of credential/selection logic. **Two caveats documented (not solved in code):** (a) macOS subscription/enterprise OAuth lives in the **Keychain (no file)** — invisible to a Linux container — so the dev runs `claude setup-token` once and exports `CLAUDE_CODE_OAUTH_TOKEN`; (b) Bedrock-via-SSO tokens expire — the host must stay `aws sso login`-fresh; the read-only mount reflects the refreshed token but the container cannot refresh it itself.

**9. Pin Node 24 in CI even though the host runs Node 25.** `package.json` engines and the CI `sidecar` job both pin Node 24. *Why:* `docs/PLAN.md` toolchain note records the host on Node 25; pinning CI to 24 avoids drift between what CI validates and the version the SDK/Fastify are targeted at, and gives a deterministic build. *Trade-off:* a minor local/CI version gap, accepted and documented.

**10. Biome (not ESLint/Prettier) + strict `tsc` + Vitest, one CI job.** Matches the repo-wide tooling decision (`CLAUDE.md` / `docs/PLAN.md §2`): Biome formatter 2-space/double-quotes/semicolons with `noExplicitAny`/`useImportType`/`noConsole: error`; strict TS (`strict`, `isolatedModules`). The `sidecar` CI job runs Biome + `tsc` + Vitest. *Why:* one tool, no split, consistent with the `web/` stream.

**11. `ai_raw` is bounded (8KB cap) with redact-then-truncate ordering — an additive safety hardening beyond the frozen contract.**
The frozen `event-envelope` contract commits only that an unknown SDK message is "never dropped, never a crash" and marks the `ai_raw` payload internals pending-spike. This change adds a deliberate safety rule on top: because `ai_raw` serializes opaque SDK content it is both an unbounded-size risk and a credential-leak vector, so the normalizer SHALL **redact first, then truncate** — redact the values of known credential-bearing keys (`api_key`/`token`/`secret`/`authorization`/password-like, case-insensitive) across the full serialized structure, and only then truncate to a fixed 8KB cap, stamping `truncated: true` when exceeded. The bearer `SIDECAR_SHARED_SECRET` and any auth tokens are never logged. *Why redact-before-truncate:* truncating first could slice the cap boundary through a key/value pair after the redactor stopped scanning, leaking a credential the redactor would otherwise have caught; redacting the full structure first makes the cap a pure size operation. *Why record it here:* this is **additive** to the frozen contract — a CHANGELOG-worthy hardening, not a contract change (it does not alter the envelope shape or the never-drop/never-crash guarantee, it constrains the `ai_raw` payload the contract left pending-spike). *Alternative rejected:* emitting the raw blob verbatim — an unbounded durable payload and a credential-leak path through the ingest log.

**12. SIGTERM does a best-effort transport flush in W1; full graceful-drain is W3; Rails (not the sidecar) finalizes an interrupted run.** On `SIGTERM`, W1 makes a single best-effort POST of any pending durable events from the transport buffer before exiting — no bounded drain window, no in-flight-run handling, no waiting; those graceful-drain semantics are deferred to W3. The sidecar never finalizes run state on shutdown or on interrupt: when a run is interrupted, the sidecar only emits the `run_interrupted` event, and **Rails** transitions an interrupted-and-dirty run to `awaiting_review`. *Why:* the W1 buffer is in-memory and lost on crash (Decision 4), so a best-effort flush on the *graceful* SIGTERM path cheaply salvages already-normalized events that idempotent ingest makes safe to re-send; full drain is unnecessary plumbing for the W1 milestone (healthz + heartbeat + ingest) and belongs with the runner/run-lifecycle work. Keeping finalize on the Rails side preserves the run-state-machine ownership boundary — the sidecar emits events, Rails owns run status — so the sidecar stays free of run-lifecycle logic in W1. *Alternative rejected:* a full graceful drain now — premature without the runner, and it would tempt the sidecar into finalize logic it must not own.

## Risks / Trade-offs

- **SDK-shape surprises (the top project risk).** → The normalizer is the only SDK-aware file and v1's contract is *never crash* — an unknown shape degrades to `ai_raw`, so a surprise is visible-but-survivable rather than fatal. The full mapping table waits for the spike; if the spike can't map messages to the taxonomy by Wednesday, only the *table* slips (≤2 days per `§14`), not this skeleton.
- **In-memory ring buffer lost on sidecar crash.** → Accepted for MVP. The durable recovery path is the Claude session JSONL in `~/.claude/projects/` (host can resume via `claude_session_id`), not the buffer; idempotent ingest makes any re-send safe.
- **Stub routes could be mistaken for working endpoints.** → Stubs return an explicit not-yet-implemented response and are documented as Week-2 fill-ins; `/healthz` is the only functional route, and the W1 milestone only depends on healthz + heartbeat + ingest.
- **Auth caveats are environmental, not code.** → Keychain-OAuth and Bedrock-SSO-expiry are documented host responsibilities; the sidecar can't and must not work around them in code (doing so would mean owning/selecting a credential, which violates Decision 8). Surfaced in the runbook/README.
- **Fixture drift between raw-SDK input and `sample_run.jsonl` output.** → The normalizer test is the cross-check: it reads the raw fixtures owned here and asserts the normalized output equals the contract's `sample_run.jsonl`; a drift fails CI.

## Migration Plan

Greenfield — no migration. Rollout is additive: the `sidecar/` package and its CI job are new. Until `rails-foundation` lands `/internal/events`, the transport target can point at a stub/log sink (the stub strategy in `docs/PLAN.md §11`); once Rails is up, the `RAILS_INTERNAL_URL` callback base URL config (the sidecar→Rails direction, distinct from `SIDECAR_URL`) flips it to the real endpoint with no code change. There is nothing to roll back beyond deleting the package.

## Open Questions

- **Exact batch size / coalescing window for `transport.ts`.** The 150ms delta coalescing is fixed by the contract, but the batch flush cadence/size for durable events is a tuning parameter to confirm during the W2 live-run pass (`docs/PLAN.md §14` streaming-jank risk).
- **Whether `GET /healthz` should report sidecar→Rails connectivity in addition to `active_run_ids`.** W1 reports `active_run_ids` only (per `sidecar-protocol`); a richer healthz is a possible W2/W3 addition and is not frozen.
