## Why

Every other stream (`api/`, `sidecar/`, `web/`) develops *inside* the containerized runtime — so that runtime has to exist before stream work can begin. This is the **dev-runtime stream** (`docs/PLAN.md §10`: "Docker Compose scaffold + `bin/start` … so every stream develops in containers from the start") and the runtime decision is final (`docs/PLAN.md §2`: Docker Compose, one container per process, `bin/start` as the single entry). It exists to make the toolchain reproducible (Ruby/Node/PG pinned in images), to map the architecture's decoupled-lifecycle invariants cleanly onto per-service containers (sidecar restarts independently of Rails; only Rails publishes a port), and to wire the host's existing Claude login into the sidecar so a developer's current auth "just works".

## What Changes

- **`bin/start`** — the single entry point a developer runs: `docker compose build` then `docker compose up` (with sensible flags). One command boots the whole stack.
- **`bin/setup`** — generates `SIDECAR_SHARED_SECRET` (the shared bearer secret between Rails and the sidecar) and prepares the env file. DB creation happens via the `rails` container's entrypoint, not here.
- **`docker-compose.yml`** — five services, **one process each**: `rails` (the ONLY service that publishes a port, `3000:3000` → LAN; Puma binds `0.0.0.0` inside the container; serves the built SPA; `depends_on` postgres healthy), `jobs` (Solid Queue supervisor `bin/jobs`, same image as `rails`, no published port), `postgres` (PostgreSQL 18, data in a **named volume**, healthcheck), `sidecar` (Node 24, port `8787` **unpublished**, reachable from `rails` as `http://sidecar:8787`, its OWN restart policy — not a child of `rails`), and `vite` (dev-only, Node 24, port `5173` **unpublished**; in the normal flow `rails` reverse-proxies the SPA + HMR to `vite`, while `vite`'s own `/api` + `/~cable` proxy to `rails` is only for running Vite directly on the host). Source is bind-mounted `:delegated`; gems and `node_modules` live in **named volumes** (no host pollution).
- **`docker/` Dockerfiles + entrypoints** — one Dockerfile per image (`rails`/`jobs` share one, `sidecar`, `web`), pinning **Ruby 4.0.5** and **Node 24** in the images. Entrypoints handle bundle/yarn install and wait-for-postgres where relevant.
- **`.dockerignore`** — excludes `.git`, `node_modules`, `vendor/bundle`, logs, `tmp`, and build artifacts.
- **Claude auth passthrough (this change OWNS the wiring)** — the `sidecar` container **read-only** bind-mounts the host `~/.claude` and `~/.aws` (the two credential dirs only — read-only for tamper-safety) while the **target repo is bind-mounted read-write** (Claude edits files in the worktree `cwd`, and a reject runs `git reset --hard HEAD && git clean -fd` in the worktree — both require a writable repo mount), and inherits the host's Claude/AWS auth env (pass-through only when set: `ANTHROPIC_API_KEY`, `ANTHROPIC_AUTH_TOKEN`, `CLAUDE_CODE_OAUTH_TOKEN`, `CLAUDE_CODE_USE_BEDROCK`, `CLAUDE_CODE_USE_VERTEX`, `AWS_PROFILE`, `AWS_REGION`, `ANTHROPIC_MODEL`) so the dev's existing login works auth-method-agnostically. Two documented host caveats: (a) macOS subscription/enterprise OAuth lives in the **Keychain (no file)** → the dev runs `claude setup-token` once and exports `CLAUDE_CODE_OAUTH_TOKEN`; (b) Bedrock-via-SSO tokens **expire** → the host stays `aws sso login`-fresh (the read-only mount reflects it; the container can't refresh itself). Mounts are read-only so a run can't tamper with host login state.
- **Worktree-path consistency** — git worktrees record **absolute** `.git` paths, so the target repo is bind-mounted at the **same path** in both the `rails` and `sidecar` containers (Rails creates worktrees at `<repo>/.clawdparty/worktrees/session-<id>`; the sidecar uses them as `cwd`).
- **Networking invariant** — only `rails` publishes a port; `sidecar` + `vite` are unpublished (the Docker equivalent of loopback-only); nothing assumes a fixed host (`SIDECAR_URL` configurable) so Tailscale is a future drop-in.

This change provides the **runtime** the sibling changes run in (`rails-foundation`, `sidecar-foundation`, `web-scaffold`). It builds compose/Dockerfiles/entrypoints/`bin` scripts only — **not** the Rails/sidecar/web application code.

## Capabilities

### New Capabilities
- `compose-stack`: The five services (`rails`/`jobs`/`postgres`/`sidecar`/`vite`), one-process-per-container, `depends_on` + healthchecks, named volumes for dependencies + postgres data, and `:delegated` bind-mounted source.
- `dev-entrypoint`: `bin/start` as the single entry point + `bin/setup` (`SIDECAR_SHARED_SECRET` generation), the per-service Dockerfiles + entrypoints, toolchain pinning (Ruby 4.0.5 / Node 24 / PostgreSQL 18), and `.dockerignore`.
- `claude-credential-mounts`: The read-only `~/.claude` + `~/.aws` mounts and the auth-method-agnostic env passthrough into the `sidecar` container, the macOS-Keychain and Bedrock-SSO caveats, and the read-only tamper-safety guarantee.
- `compose-networking`: Only `rails` publishes `3000`; `sidecar`/`vite` unpublished; `SIDECAR_URL` over compose DNS; the worktree-path-consistency requirement; and the `sidecar` independent restart policy + `~/.claude`-mount session-resume survival.

## Impact

- **New files:** `docker-compose.yml`, `docker/` (Dockerfiles + entrypoints for `rails`/`jobs`, `sidecar`, `web`), `bin/start`, `bin/setup`, `.dockerignore`, and env files: `.env.example` is the **committed template** (documents the `SIDECAR_SHARED_SECRET` slot and the pass-through auth-env list, holds no secret), while `.env.local` is the **git-ignored** file `bin/setup` writes the generated `SIDECAR_SHARED_SECRET` into. Documentation targets for the operational caveats: `README.md` updates (the Bedrock/Keychain setup notes) and a new `docs/RUNBOOK.md` (the W3 home for operational caveats — the macOS-Keychain-OAuth and Bedrock-SSO host responsibilities).
- **Provides (does not implement):** the runtime that `rails-foundation` (Rails app, `bin/jobs`, DB creation via the container), `sidecar-foundation` (Node service on `:8787`, `SIDECAR_URL`, `SIDECAR_SHARED_SECRET`, host auth env it inherits), and `web-scaffold` (Vite dev server proxying to `rails`) all run inside. Those changes own their application code; this change owns the containers, images, entrypoints, mounts, network, and `bin` scripts.
- **Dependencies / pins:** Docker + Docker Compose v2; Ruby 4.0.5, Node 24 (the host runs Node 25 — pinned to 24 in images/CI to avoid drift), PostgreSQL 18; the host developer's existing Claude/AWS login on disk (`~/.claude`, `~/.aws`).
- **Out of scope:** the actual Rails/sidecar/web application code (sibling changes); HTTPS/OrbStack mode; monitoring; CI runners for Docker (the per-stream CI jobs live in their own changes). Kept strictly to local dev.

## Dependencies

`dev-docker-compose` is the **runtime** the other Week-1 changes run inside: `rails-foundation`, `sidecar-foundation`, and `web-scaffold` all run as services in this compose stack and assume it exists. It and `freeze-interface-contracts` are **independent**: `freeze-interface-contracts` produces FILES (`packages/contracts`) while `dev-docker-compose` produces the RUNTIME, and neither depends on the other — they MAY be applied in either order. Both, however, MUST precede `rails-foundation`, `sidecar-foundation`, and `web-scaffold`, which consume the frozen contracts AND run inside this compose stack. The relationship is bidirectional in one place, however — this change provides the containers/images/entrypoints/network/mounts/`bin` scripts but deliberately does **not** implement application behavior, and it leans on `rails-foundation` for the Rails-side pieces it can only *invoke*, not author:

- **`config.hosts` + cable allowed origins** (`.local`/LAN-IP) — owned by `rails-foundation`; without them, publishing `3000` to the LAN makes Rails HostAuthorization silently block `<host>.local` (proposal/design risk; task 5.5).
- **DB-prepare / migrations across the three logical databases** (primary, Solid Queue, Solid Cable) — the rails entrypoint here *runs* `db:prepare`, but the schema/migrations it runs are `rails-foundation`'s (design Decision 8; task 3.2).
- **The dev reverse-proxy middleware** that lets `rails` front the SPA + HMR on the single published port — implemented by `rails-foundation` (compose-networking "rails fronts both API and the dev SPA" requirement); the Vite side (bind-all + HMR client port) is `web-scaffold`'s, and the `VITE_USE_POLLING` var name is coordinated with `web-scaffold`.

**Apply ordering:** `freeze-interface-contracts` (files) and `dev-docker-compose` (runtime) are INDEPENDENT and MAY be applied in either order; both MUST precede `rails-foundation`, `sidecar-foundation`, and `web-scaffold`, which consume the frozen contracts AND run inside this compose stack. Once both are applied, the sibling changes' application code drops into the bind-mounted service directories. The cross-change items above are tracked so the Rails-side dependencies are not lost between change sets.
